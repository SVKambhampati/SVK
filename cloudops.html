CloudOps Monitor

A distributed cloud monitoring pipeline built on Azure that captures application telemetry in real time, streams it through Azure Event Hub, and processes it into actionable insights for observability and alerting.

Python Flask Azure Event Hub Azure Monitor Application Insights

Overview

CloudOps Monitor is a distributed monitoring pipeline built on Azure that captures application telemetry in real time and processes it into actionable insights. A Flask-based backend generates health, metrics, and error signals, which are streamed through Azure Event Hub for high-throughput ingestion. These events are consumed by downstream services for analysis, anomaly detection, and storage.

The system is designed around reliability and scalability, using event-driven architecture to decouple producers and consumers. It mirrors how production monitoring systems handle continuous telemetry, fault detection, and alerting under load.

Key Features

  • Real-time telemetry generation across health checks, metrics, and error events
  • Event-driven pipeline using Azure Event Hub for high-throughput, decoupled ingestion
  • Structured logging with a consistent event schema across all signal types
  • Simulated failure injection for testing observability coverage and alert thresholds
  • Scalable architecture with fully decoupled producers and consumers

Technologies Used

  • Backend: Python, Flask — telemetry generation and API layer
  • Streaming / Messaging: Azure Event Hub — high-throughput event ingestion
  • Monitoring: Azure Monitor, Application Insights, Log Analytics — observability and alerting
  • Storage: Azure Blob Storage — event persistence and log archival
  • Infra: Azure CLI, GitHub — provisioning and version control

Architecture

The pipeline follows a producer/consumer model: the Flask backend acts as the producer, emitting structured telemetry events (health pings, performance metrics, and simulated errors) at configurable intervals. Azure Event Hub buffers these events and delivers them to consumer groups — each responsible for a different processing concern: real-time alerting, long-term storage, and anomaly detection.

Failure injection is built in as a first-class feature, allowing the system to simulate degraded services, elevated error rates, and latency spikes — making it possible to validate that the observability layer catches and surfaces issues before they'd affect real users.

What I Learned

This project was about understanding how monitoring systems are built internally, not just what dashboards look like. Designing around event streaming forced me to think carefully about schema consistency, consumer idempotency, and what happens when parts of the pipeline fail. Building in fault injection from the start made the observability coverage genuinely testable rather than assumed.
